# Skill Evolution

Project skills evolve from evidence.

## Add Or Strengthen A Skill When

- A repeated workflow appears in at least two real examples.
- A constraint prevents a concrete production risk.
- A rule improves signal without hiding important findings.

## Modify A Skill When

- Users misunderstand the output or workflow.
- A fixture shows the wording or process is ambiguous.
- A better local pattern emerges from implementation.

## Delete Or Weaken A Skill When

- It creates noise without improving review quality.
- It conflicts with the project positioning.
- It duplicates shared OPT behavior without project-specific value.

## Review Cadence

Review `ops/constraints` and `ops/skills` before each release and after every real-user feedback batch.
